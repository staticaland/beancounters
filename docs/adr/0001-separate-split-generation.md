# Generate split adjustments separately from imports

Split annotations are preserved inline on imported transactions, but split adjustments are generated by a separate command after import. The split generator receives an explicit config ledger plus explicit input ledger files instead of walking `main.beancount` includes automatically. Split people are defined in the config ledger with minimal `custom "split-person"` directives that map a case-insensitive key to an opened receivable account, and generated split adjustments use deterministic Beancount links to point back to the original imported transactions. This keeps monthly bank and credit-card imports disposable and debuggable while allowing `generated/YYYY-splits.beancount` to hold derived receivable adjustments for the reporting year.

Annotation preservation is implemented in Beancounters as a wrapper around fresh importer output, not inside each provider importer. This keeps reusable importer packages focused on source extraction and classification while centralizing project-specific split metadata preservation.

`preserve-splits` and `generate-splits` remain separate primitive commands. A later convenience command may wrap the full workflow, but preservation and adjustment generation are distinct steps with distinct validation and failure modes.

Preservation uses the same identity scheme as split links: provider transaction ID when available and import fingerprint otherwise. Orphaned old split annotations produce warnings, while ambiguous matches fail rather than guessing.

Generated split adjustments include deterministic links and source identity metadata by default, with an option to omit source metadata later if the output feels too noisy. Both `preserve-splits` and `generate-splits` support check modes so annotation carry-forward and split validation can run without writing ledger output.

An interactive or query-based helper for adding split annotations is deferred. Hand-editing split metadata is acceptable for the first version while the accounting model and syntax settle.

The preservation implementation should use semantic parsing for identity and validation while preferring minimal text changes when writing preserved metadata back into fresh importer output. Full AST printing is acceptable if text-level insertion proves too brittle.
